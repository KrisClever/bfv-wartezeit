1. Einleitung und Zielbild

Der Post SV Nürnberg betreibt eine Fußballabteilung mit aktuell 46 Mannschaften an zwei Standorten: Ebensee (Ziegenstr. 110, 90482 Nürnberg) und Schweinau (Daimlerstr. 71, 90441 Nürnberg). Die Belegungsplanung erfolgte historisch in Excel, wurde zeitaufwändig als PDF exportiert und manuell via WhatsApp verteilt.

Zielbild: Der Verein verwaltet Ressourcen — das System entscheidet automatisiert nach vordefinierten Regeln, sodass Freigeber nur noch bei echten Konflikten eingreifen müssen.

5a. Datenvalidierung & Edge Cases v2.1

Diese Regeln erzwingt das Backend automatisch bei jeder Buchung. Sie verhindern logische Inkonsistenzen bei der Materialisierung von Serien.

5a.1 „1-Minute-Lücke"-Regel

Direkt aufeinanderfolgende Buchungen auf demselben Feldsegment sind systemseitig zulässig. Das System empfiehlt Trainern einen 5-Minuten-Puffer, blockiert aber nicht autonom.

Team A bis 18:30 → Team B ab 18:30  ✓ zulässig
                               + Empfehlung: "5 Min Puffer eintragen?"

5a.2 Überlappende Teilbarkeiten (Segment-Sperre)

Gebuchtes SegmentAutomatisch gesperrt
Großfeld (voll)half, third, quarter derselben facilityId
Großfeld (½)Gesamtes Großfeld (voll)
Kompaktfeld (voll)2/3 des Platzes gesperrt; restliches ⅓ → Status „Eingeschränkt (nur Kleingruppen ohne Tor)"
onBookingConfirmed(booking):
  if booking.fieldConfiguration === "full":
    blockSegments(facilityId, date, start, end, ["half","third","quarter"])
  if booking.fieldConfiguration === "compact":
    markSegmentRestricted(facilityId, "third", "Nur Kleingruppen ohne Torbedarf")
  if booking.fieldConfiguration in ["half","third","quarter"]:
    blockSegments(facilityId, date, start, end, ["full"])

7.2 BFV-Spielplan-Import + Fuzzy-Matching v2.1

Der BFV liefert Spielpläne in unstrukturierten Excel-Tabellen mit abweichenden Schreibweisen. Version 2.1 löst das mit Fuzzy-Matching.
MechanismusBeschreibungSystemverhalten
Vereinsnamens-FilterZeilen ohne „Post SV Nürnberg" werden ignoriertPerformance-schonend
Levenshtein Fuzzy-Matching„Post SV Nürnberg II", „SG Post SV / Altenfurt" etc. werden erkannt≥85% → automatisch · <85% → WARNUNG + manuelle Zuordnung
Mapping-CacheManuelle Zuordnungen werden gespeichertFolge-Importe funktionieren automatisch
fuzzyMatchTeam(importedName, teams):
  if teams.find(t => t.name === importedName): return exact match
  if !importedName.includes("Post SV"): return null  // ignorieren
  best = teams.maxBy(t => levenshteinSimilarity(importedName, t.name))
  if best.score >= 0.85: return { team: best, confidence: best.score }
  return { status: "WARNUNG", confidence: best.score }

9.1 Konfliktlösungs-Matrix bei Prioritäts-Gleichstand v2.1

Wenn zwei Anfragen dieselbe Priorität haben, entscheidet das System automatisch nach dieser 3-Stufen-Kaskade.
StufeKriteriumSystementscheidung
1Stammplatz-RechtDas Team, dessen homeFacilityId NICHT mit dem angefragten Platz übereinstimmt, verliert. Heimteams haben stets Vorrang.
2Historische Zuteilung (4 Wochen)Das Team, das seltener auf dem Wunschplatz zugeteilt wurde, erhält den Zuschlag (Fairness-Algorithmus via Audit-Log).
3Zeitstempel (FiFo)Bei identischer Historie entscheidet der exakte Zeitpunkt des Buchungseingangs.
resolveEqualPriorityConflict(reqA, reqB):
  // Stufe 1: Stammplatz
  if isHomeTeam(reqA.teamId, facilityId): return APPROVE reqA
  if isHomeTeam(reqB.teamId, facilityId): return APPROVE reqB
  // Stufe 2: Historische Häufigkeit (letzte 4 Wochen)
  freqA = countBookingsLast4Weeks(reqA.teamId, facilityId)
  freqB = countBookingsLast4Weeks(reqB.teamId, facilityId)
  if freqA != freqB: return APPROVE min(freqA, freqB)
  // Stufe 3: Zeitstempel
  return reqA.createdAt < reqB.createdAt ? APPROVE reqA : APPROVE reqB
